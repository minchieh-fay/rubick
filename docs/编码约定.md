# Rubick 编码约定

本文档记录当前项目的编码偏好。这里不是泛泛而谈的最佳实践，而是项目明确希望长期坚持的写法。

## 1. HTTP 模块按业务域独立成目录

一个 HTTP 业务模块尽量放在一个独立文件夹中。

例如 `auth` 模块：

```text
auth/
├── login.ts
├── logout.ts
├── refresh.ts
└── help.ts
```

说明：

- `/auth/login`、`/auth/logout`、`/auth/refresh` 都尽量放在 `auth/` 目录里
- 不要把同一个业务域拆散到多个外部目录
- 这样更方便独立开发、独立迁移、独立删除、独立排查问题

这个规则客户端和服务端都适用。

## 2. 模块尽量自包含

无论是客户端还是服务端，模块都应优先依赖模块内部文件。

推荐原则：

- 一个模块能在自己目录内解决的问题，就不要依赖模块外文件
- 只有极少数稳定的公共类型、协议、基础工具，才放到共享层
- 业务模块之间尽量少互相引用

这样做的目的，是保持模块独立性，避免业务边界变脏。

## 3. 主业务文件保持干净

主业务逻辑应写在对应的业务文件里。

例如：

- `auth/login.ts`
- `auth/logout.ts`

这些文件应该直接体现主流程，例如：

- 参数接收
- 权限判断
- 主业务编排
- 返回结果

阅读主业务文件时，应该尽量一眼看懂这条业务链路。

## 4. 辅助逻辑统一放 `help.ts`

凡是不需要在主流程中展开阅读的辅助逻辑，统一放在模块自己的 `help.ts` 中。

例如：

- JWT 签名和校验
- Token 解析
- 字段格式化
- 复用的小工具函数
- 主流程里不需要重点阅读的辅助类方法
- 为了使用成员变量而拆出去的辅助类逻辑

这里不是说“业务类本身都应该放进 `help.ts`”。

更准确地说：

- 如果 `auth/login` 这样的主业务文件里有一个类
- 这个类里又有一些只是辅助流程的成员函数
- 这些成员函数为了访问成员变量，不方便改成普通函数

那么这部分“不需要重点阅读”的辅助类逻辑，也可以一起放到 `help.ts` 中。

这样做的目的，是让人或者 AI 在阅读代码时，优先看到主业务流程，而不是被一堆辅助函数、辅助方法打断。

## 5. 推荐示例

```text
server/src/modules/auth/
├── .env
├── index.ts
├── login.ts
├── logout.ts
├── help.ts
└── auth.sqlite
```

或者客户端：

```text
client/server/src/auth/
├── login.ts
├── logout.ts
└── help.ts
```

## 6. 例外原则

如果某个模块变得非常复杂，可以继续在模块目录内部拆更多文件，但仍建议遵守下面两条：

1. 主业务入口文件保持清晰。
2. 辅助逻辑优先在模块内部消化，不要散落到模块外。
