# 编码约定

## TypeScript 规范

### 类型优先

- 所有公开 API 必须有类型定义
- 使用 interface 定义对象结构，type 定义联合/交叉类型
- 避免 any，使用 unknown 替代

### 导出规范

```typescript
// index.ts - 统一导出
export * from './types';
export * from './implementation';

// types.ts - 类型定义
export interface ToolDefinition { ... }

// implementation.ts - 实现
export function createTool(...): ToolDefinition { ... }
```

### 命名约定

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | kebab-case | `file-reader.ts` |
| 目录名 | kebab-case | `requirement-analysis/` |
| 类名 | PascalCase | `AgentRuntime` |
| 接口名 | PascalCase | `ToolDefinition` |
| 变量/函数 | camelCase | `executeCLI` |
| 常量 | UPPER_SNAKE_CASE | `MAX_TIMEOUT` |

## 插件开发规范

### Tool 插件

每个 Tool 必须：

1. **导出默认 ToolDefinition**
   ```typescript
   const myTool: ToolDefinition = { ... };
   export default myTool;
   ```

2. **实现 execute 方法**
   - 返回 `ToolResult` 对象
   - 包含 exitCode, stdout, stderr, success

3. **错误处理**
   - 捕获所有异常
   - 返回有意义的错误信息
   - 不抛出未捕获的异常

4. **独立性**
   - 不依赖其他 tool 的内部实现
   - 通过注册表获取其他 tool
   - 配置通过参数传入

### Skill 插件

每个 Skill 必须：

1. **包含 SKILL.md**
   - 清晰的职责描述
   - 使用说明
   - 何时使用

2. **单一职责**
   - 一个 skill 只做一件事
   - 复杂功能拆分为多个 skill

## 代码组织

### 文件层级

- 主业务逻辑 → 主文件
- 辅助函数/类 → `help.ts`
- 类型定义 → `types.ts`
- 常量配置 → `constants.ts`

### 模块独立性

每个模块（core/、builtin/、custom/）应：
- 可独立编译/测试
- 最小化跨模块依赖
- 通过定义的接口通信

## Git 提交规范

### 提交信息格式

```
<type>: <description>

[optional body]
```

### Type 类型

| Type | 说明 |
|------|------|
| feat | 新功能 |
| fix | 修复 bug |
| refactor | 重构（非功能变更） |
| docs | 文档更新 |
| style | 代码格式（不影响逻辑） |
| test | 测试相关 |
| chore | 构建/工具配置 |

### 示例

```
feat: add file-reader tool
fix: handle timeout in CLI wrapper
docs: update architecture diagram
```

## 测试要求

- 每个 tool 必须有单元测试
- 核心逻辑必须有测试覆盖
- 测试文件与源文件同目录，`*.test.ts` 后缀

```
builtin/tools/file-reader/
├── index.ts
└── index.test.ts
```
