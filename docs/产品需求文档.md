# 产品需求文档 (PRD)

## 一、产品定义

### 1.1 产品名称
**Rubick** - 个人 AI 全能助手

### 1.2 产品定位
一个以 **Kanban 看板** 为核心界面的个人 AI 助手。用户通过自然语言提出任何需求，AI 自动分析、拆解任务、执行、反馈。

**核心理念：**
- 用户只管说，AI 负责做
- 任务可视化，进度一目了然
- 越用越聪明：AI 记住用户偏好、积累工具库、优化拆解策略

### 1.3 与竞品的差异

| 产品 | 形态 | 我们的差异 |
|------|------|-----------|
| Cursor | IDE 内嵌聊天 | 我们独立运行，不限于编码 |
| Notion AI | 文档编辑器 | 我们有任务流，不只是写文档 |
| Linear | 项目看板 | 我们有 AI 自动拆解和执行 |
| CoWork | 聊天为主 | 我们用 Kanban 可视化任务流 |

**我们的独特价值：Kanban + AI 自动执行 = 看得见的 AI 助手**

## 三、核心用户流程

### 3.1 用户提出需求 → AI 自动拆解

```
1. 用户在底部输入框输入需求
2. AI 分析需求：
   a. 如果信息不足 → 在聊天区追问
   b. 如果信息充分 → 创建任务卡片
3. AI 拆解子任务：
   a. 在 Inbox 创建主任务卡片
   b. 分析依赖关系
   c. 生成子任务卡片（关联到主任务）
   d. 卡片移到 Todo 列，等待用户确认
4. 用户确认后：
   a. AI 开始执行子任务
   b. 卡片移到 Doing 列
   c. 实时在聊天区更新进度
5. 完成后：
   a. 卡片移到 Done 列
   b. AI 总结完成内容
```

### 3.2 AI 能力不足时的处理

```
AI 发现缺少工具/技能：
  1. 卡片标记为 🚫 Blocked
  2. 在聊天区告知用户："我需要一个 XXX 工具来完成这个任务"
  3. 自动创建 "需要 XXX 工具" 的任务卡片
  4. 人类可以选择：
     a. 让 AI 尝试自己开发这个工具
     b. 人类自己开发
     c. 暂时跳过
```

### 3.3 越用越好用的机制

#### 3.3.1 工具库自动积累
- AI 每次开发的新工具，自动注册到 `custom/tools/`
- 下次遇到类似需求，优先查找已有工具
- 工具使用次数统计，高频工具优先展示

#### 3.3.2 用户偏好学习
- 记录用户对任务的调整（重排序、合并、拆分）
- 学习用户的任务拆解偏好
- 优化 AI 拆解策略

#### 3.3.3 模板积累
- 完成的任务可以保存为模板
- 类似需求出现时，推荐历史模板
- 模板市场（未来功能）

#### 3.3.4 技能自动升级
- 每次 Skill 执行后，AI 反思效果
- 自动优化 SKILL.md 中的指令
- 记录最佳实践

## 四、数据模型

### 4.1 Task 任务

```typescript
interface Task {
  id: number;              // 自增 ID
  title: string;           // 标题
  description: string;     // 描述
  status: 'inbox' | 'todo' | 'doing' | 'done';
  parentId: number | null; // 父任务 ID（子任务）
  color: string;           // 颜色标签 (#RRGGBB)
  tags: string[];          // 标签
  estimatedTime: string;   // 预估时间
  actualTime: string;      // 实际耗时
  createdAt: Date;
  updatedAt: Date;
  completedAt: Date | null;
  order: number;           // 列内排序
  isBlocked: boolean;      // 是否阻塞
  blockReason: string;     // 阻塞原因
}
```

### 4.2 Activity 活动记录

```typescript
interface Activity {
  id: number;
  taskId: number;          // 关联任务
  type: 'ai_message' | 'user_message' | 'status_change' | 'tool_executed';
  content: string;         // 内容
  createdAt: Date;
}
```

### 4.3 Tool 工具注册

```typescript
interface ToolMeta {
  name: string;
  description: string;
  source: 'builtin' | 'custom' | 'ai_generated';
  usageCount: number;      // 使用次数
  lastUsed: Date;
  createdAt: Date;
}
```

### 4.4 存储方案

**开发阶段：** SQLite（简单、零配置）
**未来扩展：** 可切换到云端数据库

```

└── rubick.db          # SQLite 数据库文件
```

## 七、风险与应对

### 7.1 技术风险

| 风险 | 影响 | 应对 |
|------|------|------|
| AI 拆解任务不准确 | 用户体验差 | 允许用户手动调整子任务，记录反馈优化 |
| 工具执行失败 | 任务阻塞 | 明确的错误处理，让用户选择重试/跳过 |
| Electron 包过大 | 分发困难 | 按需加载，精简依赖 |
| SQLite 并发问题 | 数据不一致 | 使用同步 API，单线程写入 |

### 7.2 产品风险

| 风险 | 影响 | 应对 |
|------|------|------|
| 用户不知道能做什么 | 上手困难 | 提供引导教程，展示可用工具列表 |
| AI 过度执行 | 用户失去控制感 | 关键操作需用户确认 |
| 任务越来越多 | 看板混乱 | 归档功能，搜索过滤 |

## 九、设计决策记录

### 为什么用 Kanban 而不是纯聊天？

1. **可视化进度**：用户一眼能看到所有任务和状态
2. **控制感**：用户可以拖拽调整，不是被动等待
3. **成就感**：卡片从左到右移动，有完成的满足感
4. **多任务管理**：同时处理多个需求不混乱

### 为什么用 SQLite 而不是 JSON 文件？

1. **查询能力**：搜索、过滤、统计
2. **数据完整性**：事务保证
3. **扩展性**：未来支持多用户/同步
4. **零配置**：单文件，无需服务器

### 为什么前端用 React 而不是其他？

1. **生态**：拖拽库、UI 组件最丰富
2. **人才**：最容易找到参考
3. **Electron**：与 Electron 集成最好
