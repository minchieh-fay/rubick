# Rubick AI 协作规范

本文档是本项目给 AI 编码工具的统一协作说明。`AGENTS.md`、`CLAUDE.md`、Cursor、Trae、Qoder 的规则文件都应与本文件保持一致。

## 项目定位

Rubick 是一个带后端服务的客户端 work app。

当前目标不是单纯做一个网页，而是做一个类似协同工作台、可逐步演进为桌面 App 的客户端产品。它包含：

- 服务端：负责能力适配、模块加载、统一接口
- 客户端页面：负责 UI 和交互
- 客户端本地后台：负责 localhost API、本地配置、本地任务和远端协调

## 开发前必须理解的上下文

AI 在开始实现前，应先理解以下事实：

1. 本项目是 Bun + TypeScript 工程。
2. 本项目同时包含服务端与客户端，不要把它当成单纯前端项目。
3. 客户端不是纯静态页面，它有 `client/server/` 这层本地后台。
4. 服务端和客户端都强调模块独立性。
5. 核心模型适配能力（ASR, TTS, Chat 等）应放在 `server/src/core/providers/` 下，业务逻辑放在 `server/src/modules/` 下。
6. 每个可独立运行的模块目录（如 `core/providers` 下的各子目录、`modules` 下的各子目录）都必须包含一个 `api.md` 文件。
7. 后续可能打包成桌面 App，所以目录与职责拆分要为未来演进留余地。

## 编码与目录规则

AI 必须遵守以下规则：

1. 一个 HTTP 业务模块尽量放在一个独立文件夹中。
2. 同一业务域的路由与逻辑尽量放在同一个模块目录，不要拆散到模块外。
3. 模块优先使用自己的内部文件，尽量保持自包含。
4. 主业务逻辑写在主业务文件中。
5. 不需要重点阅读的辅助逻辑统一收敛到模块自己的 `help.ts`。
6. 每个模块必须维护自己的 `api.md`，描述模块设计、作用及对外提供的 API 详细说明。
7. 如果主业务类中有一些只是为了访问成员变量而存在的辅助方法，也可以放到 `help.ts`，避免打断主流程阅读。
8. 服务端模块自己的 `.env`、sqlite 文件和业务实现都尽量放在模块目录里。

如果要理解这些规则的背景，优先阅读：

- `docs/编码约定.md`
- `docs/模块开发规范.md`

## 架构与目录变更规则

如果任务涉及以下内容，AI 不能只改代码不改文档：

- 框架选型变更
- 工程目录调整
- 服务端/客户端职责边界变化
- 模块规范变化
- API 调用关系变化
- 设计原则变化

只要发生上述变化，AI 必须在同一次任务中同步更新相关文档，再结束任务。

最少需要检查这些文档是否需要同步：

- `docs/目录结构说明.md`
- `docs/总体架构说明.md`
- `docs/编码约定.md`
- `docs/模块开发规范.md`
- `docs/通信与调用说明.md`
- `README.md`

## 文档更新要求

当变更影响设计或结构时，AI 应：

1. 先判断变更影响哪些文档。
2. 修改代码后，立即同步更新相关文档。
3. 保证文档中的路径使用相对路径或普通仓库路径写法，不要写死本机绝对路径。
4. 不要让文档与当前代码结构长期不一致。

## 修改原则

AI 应优先做增量修改，不要随意推翻已有设计。

如果需要做框架性调整或目录大改，应先：

1. 说明为什么要改
2. 说明会影响哪些目录或文档
3. 如果存在多种方案，优先给出方案并等待确认

## 输出要求

AI 在完成任务时，应尽量说明：

- 改了哪些代码
- 改了哪些文档
- 如果本次改动涉及架构或目录，明确说明已同步更新文档
