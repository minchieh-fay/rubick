# 统一规范

## 代码组织原则

- **可读性优先**：代码可读性高于性能等其他指标
- **单一职责**：每个模块、每个文件只做一件事，专注自己的逻辑
- **辅助函数外置**：将工具性质的辅助函数抽到独立的 `*_help.ts` 文件中，保持主代码简洁
  - 示例：`app/` 文件夹下有 `create.ts`、`modify.ts`，辅助函数（如 zip 压缩、版本号对比等）放到 `app/app_help.ts`
  - 类成员函数如果是纯工具性质，也可以抽出到 help 文件
  - help 文件放"给 AI/人看"的主逻辑和"不用细看"的工具函数之间的分界线

## 版本规范

- app/skill/mcp 全部使用语义化版本（SemVer）
- 禁止覆盖发布；只有"同一标识"的更新才允许版本号递增
- 如果只是短名相同（如都叫 `login`）但用途不同，必须视为两个独立包，不得强行升成同一包的 v2
- 版本仅表示同一包的演进，不用于解决重名问题

## 包名规范

格式：`{type}-{descriptive-name}-{YYYYMMDDHHmmss}-{version}.zip`

示例：

- `app-根据md制作ppt-20260424153000-1.0.0.zip`
- `app-根据pdf制作ppt-20260424153001-1.0.0.zip`
- `skill-linux-host-ssh登录-20260424153002-0.1.0.zip`

命名要求：

- `descriptive-name` 必须准确描述功能，不能是简单缩写（如用"根据md制作ppt"而非"ppt"）
- 时间戳精确到秒，确保同一目录下文件名不冲突
- hub 内部唯一键使用 `descriptive-name`，包名仅用于文件存储和展示
- 同一 `descriptive-name` 的更新允许版本号递增，不允许覆盖旧版本
- 如果功能不同但描述相似，必须视为两个独立包，不得强行升成同一包的 v2

## UI 规范

- `app`、`rubicktool`、`rubick` 保持统一设计语言
- 基于现有 `yy/frontend-design/SKILL.md` 增补：
  - 固定色板
  - 字体规范
  - 间距/圆角/边框层级
  - 表单控件一致性

## 调试与构建

- 调试阶段不启用 `electrobun`，保持浏览器 BS 结构便于调试
- 构建阶段加入 `BrowserWindow()` 函数，走 `electrobun` 打包桌面应用
