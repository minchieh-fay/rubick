# Rubick 模块开发规范

本文档定义 Rubick 的模块开发约定，重点覆盖服务端模块和 HTTP 相关模块的组织方式，保证后续新增功能时，目录、入口、配置、辅助代码位置都保持一致。

## 设计目标

服务端模块需要同时满足两种运行模式：

1. 被主工程自动扫描并加载。
2. 独立运行，方便单模块调试和开发。

因此每个模块都必须具备清晰的入口文件和自包含目录。

另外，本项目的模块规范不仅适用于服务端，也适用于客户端中具有明确业务边界的 HTTP 模块。

## 核心编码偏好

以下规则属于当前项目的明确编码偏好，客户端和服务端都应尽量遵守：

1. 一个 HTTP 业务模块尽量放在一个独立文件夹中，不要把同一业务拆散到模块外多个目录。
2. 模块应尽量自包含，优先使用模块内文件，不要随意依赖模块外部文件。
3. 主业务逻辑写在对应的业务文件里，辅助函数统一收敛到该模块自己的 `help.ts`。
4. `help.ts` 可以放普通函数，也可以放辅助类，目的是让主业务文件保持干净、易读。

## HTTP 模块独立性约定

无论是服务端还是客户端，只要是一个完整的 HTTP 业务域，就应尽量以“一个模块目录”承载。

例如 `auth` 模块，推荐把下面这些能力都放在同一个目录中：

- `/auth/login`
- `/auth/logout`
- `/auth/refresh`
- `/auth/profile`

推荐结构如下：

```text
auth/
├── login.ts
├── logout.ts
├── refresh.ts
├── profile.ts
└── help.ts
```

设计原则：

- 路由属于哪个业务域，就尽量放在该业务域目录内。
- 不要把 `login.ts` 放一个目录、`logout.ts` 又放另一个公共目录。
- 除了非常稳定的公共协议或基础类型，不要直接依赖模块外部文件。
- 模块迁移、删除、单独调试时，应尽量只处理该模块目录本身。

这个规则在 `server` 和 `client` 两侧都成立。

## 模块目录约定

推荐结构如下：

```text
server/src/modules/<module-name>/
├── .env
├── index.ts
├── help.ts
├── services/
├── adapters/
├── schemas/
└── <module-name>.sqlite
```

说明：

- `.env`
  模块自己的环境配置。
- `index.ts`
  模块入口文件，负责导出模块定义。
- `help.ts`
  模块自己的辅助函数文件。主流程里不需要展开的杂项逻辑、工具函数、辅助类，优先放在这里。
- `services/`
  模块内部服务实现。
- `adapters/`
  外部能力适配层，例如第三方模型 SDK、外部 API。
- `schemas/`
  请求参数、响应结构或模块内部数据结构定义。
- `<module-name>.sqlite`
  模块自己的 sqlite 文件。只有需要数据库时才创建。

## 模块职责边界

一个模块应只关心自己的业务能力，不应该直接承担其他模块的职责。

例如：

- `chat` 模块只负责聊天相关能力。
- `asr` 模块只负责语音转文本。
- `tts` 模块只负责文本转语音。
- `embd` 模块只负责向量化或 embedding 能力。

如果多个模块共享某些极稳定的工具或类型，应放到 `shared/` 或公共层；除此之外，优先保持模块内自给自足，而不是互相穿透依赖。

## 主业务文件与 `help.ts` 约定

你希望主业务代码尽量干净，这里把规则明确化：

### 主业务文件

主业务文件负责承载：

- 路由入口
- 请求参数接收
- 主流程编排
- 关键业务判断
- 返回结果组织

例如：

- `auth/login.ts`
- `auth/logout.ts`

这些文件应该让人一眼就看出主流程，不要被大量细枝末节打断。

### `help.ts`

`help.ts` 负责承载主流程之外的辅助逻辑，例如：

- JWT 签名和校验
- Token 提取
- 字段格式转换
- 一些复用的小型校验逻辑
- 主流程里不需要展开阅读的辅助类

目标不是追求“绝对纯净分层”，而是优先让主业务文件易读。

例如：

```text
auth/
├── login.ts
├── logout.ts
└── help.ts
```

其中：

- `login.ts` 专注登录主流程
- `logout.ts` 专注退出主流程
- `help.ts` 放签名、解签、字段处理、辅助类等杂项能力

如果某个模块后续足够复杂，也可以在保留 `help.ts` 的前提下，再拆更多子文件，但主业务文件保持干净这一原则不变。

## 模块入口约定

模块入口文件应满足以下要求：

1. 默认导出模块定义对象。
2. 定义自己的基础信息，例如名称、路由前缀、描述。
3. 支持被主工程加载。
4. 支持独立运行调试。

当前建议的入口职责：

- 暴露模块元信息
- 注册模块路由
- 初始化模块资源
- 在独立运行模式下启动自己的调试服务

## 配置约定

每个模块的环境变量应该尽量只服务于该模块自身，不要把模块私有配置堆到全局根目录。

例如：

```env
PORT=3100
MODEL_PROVIDER=openai
MODEL_NAME=gpt-4.1
```

如果后续存在全局配置，再单独设计服务端根级配置文件；模块私有配置仍建议保留在模块目录内。

## 数据约定

模块如果需要落盘数据，应优先存放在自己的目录内。

推荐原则：

- sqlite 文件放在模块目录本身
- 模块缓存文件放在模块目录本身
- 模块迁移脚本、初始化数据也尽量跟模块放在一起

这样做的好处是：

- 模块迁移更简单
- 模块复制或裁剪更容易
- 模块的配置和数据边界更清晰

## 路由约定

模块应自行声明自己的路由前缀，例如：

- `/chat`
- `/asr`
- `/tts`
- `/embd`

主工程负责统一加载模块，但不替模块决定业务路由路径。

## 建议的开发顺序

新增一个模块时，建议按以下顺序完成：

1. 创建模块目录和 `.env`
2. 先确定该模块是否需要独立的 `help.ts`
3. 编写 `index.ts` 或主业务文件
4. 把主流程留在业务文件中，把辅助逻辑收敛到 `help.ts`
5. 实现 `services/` 内部业务逻辑
6. 补充路由注册逻辑
7. 如有需要，再创建 sqlite 文件
8. 最后补充该模块自己的说明文档
