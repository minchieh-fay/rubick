# Rubick 总体架构说明

本文档描述 Rubick 第一阶段的整体架构设计，用来回答三个问题：

1. 系统为什么拆成现在这样。
2. 服务端、客户端、共享层分别负责什么。
3. 后续功能应该优先落在哪一层。

## 总体目标

Rubick 的目标不是先堆功能，而是先建立一个适合持续扩展的 Agent 工程基础设施。

第一期重点如下：

- 服务端负责统一适配模型能力，例如 Chat、ASR、TTS、Embedding。
- 服务端采用模块化设计，每个模块独立维护自己的配置和数据。
- 客户端默认按 B/S 架构设计，但保留未来封装为桌面 App 的空间。
- 客户端页面既可以调用本地 Bun 后台，也可以调用远端服务端。
- 客户端与服务端都遵守模块独立性优先的编码风格。

## 架构分层

### 1. 服务端层

目录：`server/`

职责：

- 统一暴露 HTTP 接口。
- 自动扫描并加载业务模块。
- 维护核心供应层（Core Providers），统一适配模型能力（Chat, ASR, TTS, Embedding 等）。
- 承担模型适配和业务编排职责。
- 作为系统能力中心，对外提供稳定的 API。

这一层是能力汇聚层。后续不同模型厂商、不同能力类型的差异，应通过 `core/providers/` 目录下的适配层被吸收到系统内部。

### 2. 核心供应层 (Core Providers)

目录：`server/src/core/providers/`

职责：

- 统一封装基础模型能力。
- 对上层 `modules/` 提供一致的 AI 模型接口。
- 隐藏不同厂商、不同模型的接口差异。

这一层存放的是系统运行必须的模型能力适配器（如 Ollama, OpenAI, Azure 等）。

### 3. 客户端页面层

目录：`client/web/`

职责：

- 承载用户界面。
- 负责用户交互和页面状态管理。
- 发起对本地 API 或远端 API 的调用。
- 后续作为桌面 App 的可复用 UI 层。

这一层不应该承担复杂业务编排，也不应该直接承载本地持久化逻辑。

### 3. 客户端后台层

目录：`client/server/`

职责：

- 在本机监听 localhost 端口。
- 为前端页面提供本地 API。
- 管理客户端本地配置、缓存、会话和本地任务。
- 在需要时代理、转发或协调与服务端的通信。

这一层是客户端侧的“本地能力层”，它让前端页面保持轻量，同时也为将来打包为桌面 App 做准备。

### 4. 共享层

目录：`shared/`

职责：

- 存放前后端共用的类型定义。
- 存放通用工具函数和公共协议结构。
- 尽量避免服务端与客户端重复定义同类数据结构。

## 运行视角

从运行方式看，当前系统会有三个潜在进程：

1. 服务端进程
2. 客户端后台进程
3. 客户端页面进程

它们之间的关系如下：

```text
Browser / WebView
       |
       v
client/web
   |        \
   |         \ HTTP
   v          \
client/server  --->  server
```

说明：

- `client/web` 可以直接调用 `server`
- `client/web` 也可以先调用 `client/server`
- `client/server` 可以负责本地配置、本地缓存、本地鉴权状态和请求转发
- `server` 负责统一封装模型能力和业务模块

## 为什么这样设计

### 模型适配与业务模块分离

我们将能力分为两类：

1. **核心能力供应层 (Providers)**：存放于 `server/src/core/providers/`。如 `ollama`, `openai`, `asr`, `tts`。这些是系统运行的必须设施，为上层提供基础模型调用能力。
2. **业务功能模块 (Modules)**：存放于 `server/src/modules/`。如 `auth`, `workflow`。这些是基于核心能力实现的具体业务逻辑，具有更强的可插拔性。

两类模块都遵循同样的独立文件夹原则：拥有自己的 `.env` 和数据文件。

### 客户端分成 web 和 server

如果客户端未来要变成桌面 App，那么单纯只有前端页面通常不够，还需要一个本地后台来承接：

- 本地配置管理
- 本地数据库读写
- 文件系统访问
- 本地任务调度
- 与远端服务端的协调

因此现在直接把客户端拆成 `web` 和 `server`，后期扩展成本更低。

## 编码组织原则

除了架构分层，当前项目还强调代码组织方式保持一致。

核心原则如下：

1. 一个 HTTP 业务域尽量放在一个独立目录中。
2. 模块优先依赖自己的内部文件，保持自包含。
3. 主业务流程写在主业务文件里。
4. 辅助函数、辅助类、杂项逻辑优先收敛到模块自己的 `help.ts`。

这套规则对 `server` 和 `client` 两侧都适用。

## 后续扩展方向

建议按以下顺序继续推进：

1. 完善 `core/providers/` 内部的模型接口规范。
2. 明确业务 `modules/` 的生命周期和统一注册协议。
3. 完成服务端统一路由注册。
4. 明确客户端 `web -> server -> server` 的调用边界。
5. 为共享层补充公共类型定义。
6. 再逐步接入真实的业务模块。
