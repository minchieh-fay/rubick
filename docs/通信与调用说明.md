# Rubick 通信与调用说明

本文档说明当前工程中客户端与服务端之间的调用关系，以及后续应该如何划分接口边界。

## 当前角色

系统中当前存在三个主要角色：

1. `client/web`
2. `client/server`
3. `server`

它们不是简单的前后端二层结构，而是三段式调用关系。

## 调用关系

### 1. `client/web -> client/server`

适合承载本地相关能力，例如：

- 本地配置读取
- 本地缓存读取
- 本地会话状态
- 本机文件处理
- 桌面化后需要操作系统能力的场景

### 2. `client/web -> server`

适合承载纯远端能力调用，例如：

- 聊天请求
- 语音识别请求
- 文本转语音请求
- embedding 请求

### 3. `client/server -> server`

适合承载需要客户端后台协调的能力，例如：

- 在本地做预处理后再调用远端服务
- 统一管理服务端地址和鉴权信息
- 将多个远端请求聚合成一个本地接口

## 为什么不让 `client/web` 直接做所有事情

如果所有逻辑都直接塞到页面层，会带来几个问题：

- 页面层会变得过重
- 本地配置和本地数据难以管理
- 后续打包桌面 App 时很难平滑演进
- 前端会直接承受过多服务端细节

因此引入 `client/server` 作为本地后台，是为了把“页面交互”和“本地能力”分离开。

## 接口边界建议

建议从一开始就把接口分成两类：

### 本地接口

由 `client/server` 提供，例如：

- `/api/config`
- `/api/session`
- `/api/local-tasks`

### 远端接口

由 `server` 提供，例如：

- `/chat`
- `/asr`
- `/tts`
- `/embd`

这样做的好处是前端调用时语义清晰，不会把本地能力和远端能力混在一起。

## 配置流向建议

推荐配置流向如下：

1. `client/web` 不直接保存复杂配置
2. `client/server` 负责维护客户端本地配置
3. `client/server` 决定当前连接哪个 `server`
4. `server` 自己维护自己的服务端配置
5. 服务端模块自己维护模块级 `.env`

## 后续建议

1. 为 `client/server` 定义统一本地 API 前缀。
2. 为 `server` 定义统一健康检查和模块清单接口。
3. 明确哪些能力允许前端直连服务端，哪些必须经过客户端后台。
4. 后续如果引入鉴权，应优先放在 `client/server` 和 `server` 的边界层处理。
