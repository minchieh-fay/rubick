# 设计思想

## 项目定位

Rubick 是一个**个人 AI 助手工作台**，类似 CoWork 的概念。

核心理念：**人类只提需求和产品方向，AI 负责一切。**

人类不需要懂技术、市场、产品设计。他们只需要"唧唧歪歪"说出想法，AI 负责：
- 产品定义与设计
- 架构设计
- 开发实现
- 测试验证
- 打包部署
- 风险把控

## 架构哲学

### 1. 插件优先

一切皆插件。每个工具/技能都是独立可插拔的单元：

```
一个目录 = 一个工具
一个目录 = 一个技能
```

每个插件：
- 有独立的 `index.ts` 入口
- 能独立完成特定任务
- 能独立验证和测试
- 不依赖其他插件的内部实现

### 2. 解耦与抽象

上层调用者**不知道也不关心**底层实现：
- 只知道 `CLI` 抽象接口
- 不关心是 qodercli 还是其他工具
- 通过统一的 ToolDefinition 接口交互

### 3. 双层插件体系

```
builtin/          ← 官方维护，随项目分发，可上传 Git
custom/           ← 用户自定义，本地扩展，不进入版本控制
```

这样设计的原因：
- 官方工具保证质量和稳定性
- 用户工具满足个性化需求
- 两者互不干扰

### 4. Agent 角色分离

不同任务需要不同角色的 Agent：

```
需求分析 Agent → 需要：需求收集技能、沟通能力
K8s 运维 Agent → 需要：K8s 工具、运维技能
开发 Agent     → 需要：编码工具、测试技能
```

通过 `--agents "path/to/SKILL.md"` 加载特定角色。

## 技术选型理由

| 选择 | 理由 |
|------|------|
| Bun | 快速启动、原生 TypeScript、易分发 |
| TypeScript | 类型安全、开发体验好 |
| Hono | 轻量、高性能 HTTP 框架 |
| Electron | 跨平台桌面打包能力 |

## 开发流程

```
用户提需求
    ↓
AI 分析需求完整性
    ↓
信息不足？→ 主动询问用户
    ↓
缺少工具/技能？
    ↓
Agent 能实现？→ 自动开发
    ↓
Agent 不能实现？→ 记录为 Issue，让人类处理
    ↓
实现 → 测试 → 交付
```

## 风险把控

1. **不要盲目迎合用户** - 用户可能不懂技术，提出的方案可能不合理
2. **主动识别风险** - 安全、性能、可维护性
3. **记录决策过程** - 为什么这么做，替代方案是什么
4. **周期性总结** - 防止上下文丢失
